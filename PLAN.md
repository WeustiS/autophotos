# autophotos — Architecture Plan

A pipeline to ingest, cull, edit, and creatively display photos from a Sony A7R V.
Four stages, built to be shipped independently, unified by a single embedding backbone.

---

## 0. The unifying idea: one embedding backbone

Nearly everything below — cull scoring, burst dedup, semantic categories, the
hot↔cold display axes, and the website's semantic search — runs off **one image
embedding per photo**. Run every image through a vision-language encoder once,
cache the vectors, reuse everywhere.

- **Encoder:** OpenCLIP (ViT-L/14 or bigger) or **SigLIP / SigLIP 2** (generally
  stronger than CLIP for retrieval). SigLIP for the backbone, with a CLIP text
  tower available because the semantic-axis trick (Stage 3) needs a shared
  text-image space.
- **Storage (scale: thousands of images — keep it dead simple):**
  - Vectors: a single flat array. At a few thousand images, brute-force cosine in
    numpy is sub-millisecond — **no vector DB, no ANN index needed.** Store as one
    `embeddings.npy` (+ an `ids.json` parallel array), or one `.npy` sidecar per
    photo. Skip hnswlib/Qdrant/LanceDB entirely; they solve a scale problem you
    don't have.
  - Metadata: SQLite as a **rebuildable cache, not the source of truth** (see
    Design principles). Single file, embeddable in the Tauri app.
- **One pass, many heads.** From the cached embedding you derive: aesthetic score
  (linear head), dedup clusters (cosine), category labels (zero-shot or cluster
  + caption), and axis projections. Adding a new capability later = a new tiny
  head, not a new model run.

This is the single most important design decision: **make the embedding the
shared currency between all four stages.**

## Design principles

Two rules that keep the system simple and let you edit the filesystem by hand
without breaking anything. At a scale of *thousands*, both are easy to honor.

### 1. The filesystem is the source of truth. Everything else is a rebuildable cache.

- **Source of truth = the RAW files on disk + their XMP sidecars** (ratings,
  picks, labels live in `.xmp` next to each `.ARW`). Move, rename, copy, or delete
  files in Finder/Explorer freely — nothing breaks.
- **SQLite, the embedding array, thumbnails, captions — all derived.** Any of them
  can be deleted and regenerated by a `rescan` that walks the folder. Treat them
  as a cache you could `rm -rf` at any time.
- **Reconcile on a content hash, not a path.** Key every derived record (embedding,
  thumbnail, score) by a hash of the file's pixels/bytes, not its path. Then a
  manual move/rename is a no-op (same hash, new path → just update the path), and
  a true new file gets picked up. Deletes are detected as "hash no longer on disk."
- **Rescan is idempotent and cheap** at thousands of files — run it on app launch
  and on a folder-watch event. No "import database" that drifts from reality.

### 2. Simple, inspectable formats over clever ones.

- Sidecars in plain text/JSON/`.npy` where practical, so you can `cat`, diff, and
  grep them. Prefer one-file-per-photo sidecars (greppable, mergeable, survive
  partial corruption) over one monolithic binary blob — at thousands of files the
  overhead is irrelevant and the debuggability is worth a lot.
- The website's data export is just static JSON + thumbnails (Stage 4) — readable,
  versionable, no server.

This is why no vector DB: at thousands of images a flat `.npy` you can load,
inspect, and rebuild beats an opaque indexed store you can't hand-edit.

## Stage 1 — Ingest / view / rate / queue (Tauri app)

**Stack:** Tauri (Rust core + web frontend). Near-native speed with a web UI —
the right call for fast 100% loupe zoom on 60MP files where a pure browser app
stutters.

### The RAW speed trick

A7R V `.ARW` files are ~60MP, 60–120MB. **Do not decode RAW for browsing.** Every
ARW embeds a full-resolution JPEG preview — extract it and previews are instant.

- Rust: `rawler` or `libraw` bindings to pull the embedded preview + EXIF.
  (Python equivalents for prototyping: `rawpy`, `exiftool`, `libraw`.)
- Only run a true RAW develop (libraw / `darktable-cli`) **on demand** at the
  edit stage, never during culling.
- Pre-generate a thumbnail pyramid (e.g. 256 / 1024 / full-preview) on import;
  cache to disk keyed by file hash.

### Ratings & queue — interop is the whole game

Write ratings, picks/rejects, and color labels to **XMP sidecar files**, not just
the app DB. This is the industry standard; it means Lightroom, darktable, and
Capture One all read your ratings (and you read theirs). Mirror into SQLite for
fast querying.

- Culling UX that actually feels fast is keyboard-driven: `P`/`X` pick/reject,
  `1`–`5` stars, arrow nav, space for loupe, `Z` for 100%.
- The "edit queue" is just a saved filter/smart-collection over metadata
  (e.g. `pick == true AND edited == false`), not a separate data structure.

### Build vs. fork

- **Build (recommended):** Tauri app, your own UX, full control. Most work but
  best fit for the custom downstream stages.
- **Fork:** [digiKam](https://www.digikam.org/) (open-source Qt/C++, already has
  ratings, SQLite, face recognition) is the most forkable full app if you'd
  rather extend than start fresh.
- **Study for UX:** Narrative Select (praised as the fastest RAW-preview
  experience), Aftershoot (local, learns your style), FilterPixel, Photo Mechanic
  (speed king, closed).

## Stage 2 — Auto-cull, aesthetic scoring, crops, auto-edit

Your past approach (CNN trained on Instagram like-ratios as a quality proxy) had
the right instinct but a known failure mode — see the deep-dive below. The modern
pipeline separates **technical quality**, **general aesthetics**, and **your
personal taste**, and rides the shared embeddings.

### 2a. Technical quality (cheap, deterministic, runs first)

This is most of what "culling" actually is, and none of it needs a fancy model:

- **Sharpness / focus:** Laplacian variance, or a small learned head; flag soft
  shots and back/front-focus misses.
- **Blink / eyes-closed** and basic face checks (for portraits/people).
- **Exposure:** clipping histograms, blown highlights, crushed shadows.
- **Burst dedup:** cluster near-duplicates by embedding cosine similarity, then
  surface only the best of each burst. This is the single biggest time-saver.

### 2a-bis. Stack detection — group, don't auto-decide (IN SCOPE)

You shoot focus stacks / exposure brackets / panos. To a pure embedding
clusterer these look identical to bursts, and reliably auto-classifying *type* is
hard — so the system's job is to **propose candidate stack groups and queue them
for you**, not decide what they are. You assemble or discard by hand.

**Cheap, reliable signals (EXIF first, pixels second):**

- **Exposure brackets:** sequential frames, near-identical framing, **EV/shutter/
  aperture stepping** in a regular pattern; A7R V bracketing writes these as a run.
  High embedding similarity + varying exposure metadata = bracket.
- **Focus stacks:** the A7R V's focus-bracketing drive writes a rapid run with
  **monotonically shifting focus distance** (and often a focus-bracket tag); same
  exposure, near-identical framing. High similarity + focus-distance drift = focus
  stack.
- **Panoramas:** rapid run, **consistent exposure**, but framing/orientation
  *drifts* (lower pairwise similarity than a bracket, with a directional shift) —
  detectable as a chain of overlapping-but-sliding frames.
- **Plain burst:** rapid run, near-identical settings, near-identical framing, no
  systematic EV or focus drift → dedup as in 2a.

**Output:** a **"stacks to review" queue** (sibling of the edit queue in Stage 1).
Each candidate group shows the frames, a guessed type (bracket / focus / pano /
burst) as a *hint only*, and the EXIF that triggered it. You confirm the grouping,
pick a type/action (assemble, keep-best, discard), and the rest flows on. Keep the
grouping liberal — over-grouping is fine since you adjudicate; the cost of a
missed group is worse than a false one.

**Implementation:** group by tight timestamp runs first (drive-mode / continuous
shooting clusters), then split/merge using the EXIF deltas above, with embedding
similarity as the tiebreaker. All deterministic and cheap — no model needed for
the grouping itself.

### 2b. General aesthetics (zero-shot to start)

- **Easy baseline:** the [LAION aesthetic predictor](https://github.com/LAION-AI/aesthetic-predictor)
  — a tiny linear head on CLIP embeddings. Free, since you already have the
  embeddings.
- **Opinion-unaware:** [CLIP-IQA](https://arxiv.org/abs/2207.12396) /
  [QualiCLIP](https://github.com/miccunifi/QualiCLIP).
- **SOTA ranking:** [Q-Align](https://q-align.github.io/) (an LMM that scores
  quality *and* aesthetics via text-defined levels, strong human alignment) and
  2025 successors like VisualQuality-R1 and [UniQA](https://openreview.net/forum?id=8mE8KNHTjd).
  Use one of these to *rank* keepers after the cheap heads filter the obvious
  rejects.

### 2c. Personal taste (PIAA) — FUTURE / deferred

Not in the initial build. The plan: once enough of your star ratings have
accumulated (they're collected for free during culling), fit a **few-shot
personalized aesthetics** head on top of the frozen backbone, or **LoRA-tune** a
small adapter on the encoder/VLM for your taste. Defer until there's a corpus
worth training on — zero-shot ranking (2b) carries the early days. See the
deep-dive for the PIAA references when we pick this up.

### 2d. Crop / composition suggestions — IN SCOPE

This works well now. Pick one:

- **Baseline:** [GAIC](https://github.com/HuiZeng/Grid-Anchor-based-Image-Cropping)
  (grid-anchor cropping, pretrained weights). GAICv2 has ~236k annotations.
  Lightweight, fast, easy to integrate — good default.
- **Newer / better "why":** [ProCrop](https://arxiv.org/abs/2505.22490) (2025;
  retrieves professional compositions to guide crops + scores them).

Plan: start with GAIC for a quick win; evaluate ProCrop on your own shots and
switch if its crops feel better. Both just propose 2–3 candidate boxes per image.

### 2e. Auto-edit (RAW develop) — OUT OF SCOPE for now

Explicitly deferred. When revisited, the approach would be to predict *develop
parameters* (exposure, WB, tone, HSL) from embeddings rather than build a
pixel-to-pixel editor — but it's not part of the current build.

## Stage 3 — Semantic categories & axes

Your word2vec instinct pays off cleanly here.

### Model-consistency rule (QoL — do this from day one)

Lock **one** embedding model and **one** captioning VLM for the whole project and
never silently swap them. Everything in this stage (and the website) reuses the
exact vectors computed at import in Stage 0 — no re-embedding, no separate
training runs, no second model to keep in sync.

- The hot↔cold axes and semantic search **must** use the same text-image space
  the images were embedded in, or projections are meaningless. So whichever
  encoder you choose in Stage 0 (SigLIP / CLIP) is the one used here, period.
- Pick a single VLM for captioning/labeling (Qwen-VL or LLaVA) and pin the
  version. Captions are computed once and reused as categories, alt-text, and
  searchable metadata.
- Treat the model choice like a schema: changing the encoder later means a full
  re-embed of the library, so decide once, write it down, and only migrate
  deliberately.

### Auto-discovered categories

- **Cluster then label:** HDBSCAN/k-means on embeddings → caption each cluster
  with a VLM (Qwen-VL, LLaVA, BLIP-2) → human-readable category names.
- **Zero-shot tagging:** CLIP zero-shot against a candidate label set you curate
  (subjects, settings, lighting, mood).

### The hot↔cold / furry↔spiky axes (the fun part)

This is the elegant core of your visualization idea:

1. Embed both pole phrases as **text** ("hot" / "cold").
2. Take the **difference vector** (the axis direction in embedding space).
3. **Project** each image embedding onto it → a scalar position per photo.

You can define **arbitrary axes on the fly**, including ones invented at display
time, because CLIP's joint text-image space lets text define directions that
images live along. Same mechanism as word2vec analogies. Normalize per-axis
(z-score or rank) so layouts are well-spread.

## Stage 4 — Website display

Precompute everything offline; ship **static JSON + thumbnails**. No live model
on the site → cheap, fast, cacheable.

At thousands of images this is trivially client-side — the whole embedding set
(a few thousand × ~1k floats ≈ a few MB) ships to the browser and all
search/projection runs in JS. No server, no serverless endpoint.

- **Semantic search:** ship the precomputed text-tower embeddings for your label
  vocabulary (or compute query directions offline); cosine-rank in-browser.
- **"Galaxy" map:** UMAP/t-SNE of embeddings → a scatter of thumbnails to
  pan/zoom.
- **Custom axes:** project onto chosen pole-pairs; lay thumbnails along x/y; let
  the viewer **pick the two axes** from dropdowns — the signature visualization.
- **Rendering:** a few thousand thumbnails are fine on WebGL —
  [deck.gl](https://deck.gl/) ScatterplotLayer / custom thumbnail layer, or
  pixi/regl. Next.js shell. (Plain canvas would even cope; WebGL just keeps
  pan/zoom buttery.)

## Cross-stage integration ideas (the glue)

These are where the system becomes more than four tools:

1. **Embedding is computed once at import** (Stage 1) and flows to every later
   stage. The Tauri importer runs the encoder and writes vectors to the shared
   store immediately, so culling, scoring, and the website never recompute.

2. **Ratings are training labels (for the deferred 2c).** Every star you assign
   in the viewer (Stage 1) is a labeled example banked for the future PIAA/LoRA
   head — culling builds the dataset for free, so 2c is cheap to turn on later
   with zero extra labeling effort.

3. **Active learning loop (future, once 2c is on).** The aesthetic head proposes a
   pre-sort when you open a shoot; you correct it; disagreements feed back as the
   highest-value labels. Until then, zero-shot ranking does the pre-sort.

4. **Crop suggestions respect the edit queue.** When you pick a photo into the
   edit queue, Stage 2d generates 2–3 crop proposals on the side so editing
   starts from composition, not a blank slate.

4b. **Stacks share the burst clusterer, split by EXIF.** The same similarity
   clustering that powers dedup (2a) feeds stack detection (2a-bis); EXIF deltas
   route a cluster into either the dedup path or the "stacks to review" queue.
   One pass, two outcomes — and you adjudicate the ambiguous ones by hand.

5. **The website inherits the cull, automatically.** "Published" = a saved
   filter (e.g. `5-star AND edited`). The static-site build job reads that filter,
   pulls the already-cached embeddings + axis projections, and regenerates the
   galaxy. Publishing is a metadata flag, not an export pipeline.

6. **Axes double as cull lenses.** The same projection trick that arranges the
   website ("hot↔cold") works inside the viewer as a sort/filter — e.g. sort a
   shoot by "candid↔posed" or "minimal↔busy" while culling.

7. **One captioning pass, three uses.** VLM captions generated for categories
   (Stage 3) also become website alt-text/SEO and searchable metadata in the
   viewer.

## Reference: PIAA papers (for when 2c comes off the to-do list)

The modern toolkit just makes the personalized-aesthetics idea practical in a way
it wasn't pre-foundation-models: a frozen CLIP/SigLIP backbone + a small head (or
a LoRA adapter) learns from a handful of *your* labels, where a from-scratch CNN
on engagement data needed millions and still only learned "what spreads."

Personalized Image Aesthetics Assessment (PIAA) references for the future build:

- [Personalized Image Aesthetics Assessment with Rich Attributes (arXiv:2203.16754)](https://arxiv.org/abs/2203.16754)
- [Scaling Up PIAA via Task Vector Customization (arXiv:2407.07176)](https://arxiv.org/abs/2407.07176)
- [What VLMs Encode for PIAA (arXiv:2604.11374)](https://arxiv.org/html/2604.11374)
- [PreferThinker: reasoning-based personalized preference (arXiv:2511.00609)](https://arxiv.org/pdf/2511.00609)

Your star ratings are exactly the few-shot supervision these methods want, and
they accumulate for free during culling.

## Suggested build order

1. **Tauri culling viewer** + embedded-preview pipeline + XMP ratings + SQLite +
   the import-time embedding pass. (Produces your training data.)
2. **Embedding-backed cull:** dedup, **stack grouping → "stacks to review" queue**,
   zero-shot aesthetic ranking, and GAIC crop proposals bolted onto the viewer.
3. **Semantic axes + website:** mostly a frontend job once embeddings exist.

Deferred (collecting ratings the whole time so they're ready when we get here):
- **2c — Personal taste (PIAA / LoRA)** once a rated corpus has built up.
- **2e — Auto-edit** out of scope for now.

---

## Resolved decisions

- **Scale:** thousands → flat `.npy` embeddings + brute-force cosine, SQLite as a
  rebuildable cache, fully client-side website. No vector DB.
- **Source of truth:** filesystem + XMP sidecars; all indexes rebuildable via
  hash-keyed rescan, so manual file edits are always safe.
- **2c (PIAA/LoRA):** deferred. **2e (auto-edit):** out of scope.
- **Crops:** GAIC first, evaluate ProCrop later.

## Open questions to resolve next

- Single-user local-only, or eventual multi-device sync of ratings? (XMP sidecars
  make this a non-issue for a simple cloud-folder sync if you ever want it.)
- Folder layout convention for the library (by date? by shoot?) — affects the
  rescan/watch setup but nothing fundamental.
